## 1. 为什么需要这两门"工程"

大语言模型本质上是一个**条件概率预测器**：给定一段输入文本（上下文），它预测"下一个最可能出现的 token"。这意味着——

- 模型的输出**完全由你提供的输入决定**；
- 同样的问题，输入组织方式不同，结果质量可能天差地别；
- 模型没有"记忆"，每次调用看到的只有你这次塞进上下文窗口的内容。

因此，"如何组织输入"就成了决定 LLM 应用成败的关键。围绕这个问题，逐渐形成了两个相关但侧重不同的方向：

| 维度 | 提示词工程 | 上下文工程 |
| --- | --- | --- |
| 关注点 | **怎么问**（指令的措辞、结构、示例） | **给它看什么**（喂入哪些信息、如何检索与组织） |
| 类比 | 给员工下达一条清晰的指令 | 为员工准备好完成任务所需的全部资料 |
| 主要手段 | 角色设定、Few-shot、思维链、输出约束 | RAG、记忆管理、信息压缩、上下文窗口编排 |
| 演进关系 | 更早出现，偏"话术" | 更晚兴起，偏"系统/架构" |

> 一句话总结：**提示词工程是"把话说对"，上下文工程是"把料备齐"。** 现代 LLM 应用（尤其是 Agent）二者缺一不可。

## 2. 基础原理：LLM 是如何"思考"的

理解原理，才能解释为什么这些技巧有效。

1. **自回归生成（Autoregressive Generation）**
   模型逐 token 生成，每个新 token 都以"之前所有 token"为条件。前文质量直接影响后文。

2. **上下文窗口（Context Window）有限**
   模型一次能"看到"的 token 数量是有限的（例如 8K / 128K / 1M）。超出部分会被截断或遗忘，因此**信息的取舍与排布**至关重要。

3. **注意力机制与位置效应**
   模型对上下文不同位置的关注度不均衡，存在 **"中间遗失"（Lost in the Middle）** 现象：开头和结尾的信息更容易被利用，中间的长文本容易被忽略。

4. **概率采样与不确定性**
   输出受温度（temperature）等参数影响，具有随机性。约束越明确，输出越稳定。

5. **没有内在记忆与实时知识**
   模型不知道训练截止之后的事，也记不住上一次对话。**所有"上下文"都得由你在调用时显式提供。**

这五条原理，直接推导出后面所有技巧的"为什么"。

## 3. 提示词工程（Prompt Engineering）

### 3.1 核心思想

> **用清晰、结构化、带约束的自然语言，把任务、角色、要求和输出格式准确地传达给模型，减少歧义、引导推理。**

好的提示词通常包含以下要素（不必全有，按需组合）：

- **角色（Role）**：你是谁，以什么身份回答。
- **任务（Task）**：要做什么。
- **上下文/背景（Context）**：相关背景信息。
- **约束（Constraints）**：长度、语气、禁止项、必须项。
- **示例（Examples）**：示范期望的输入输出。
- **输出格式（Format）**：JSON、表格、Markdown、步骤列表等。

### 3.2 基本原理

- **降低歧义**：模型在概率空间里"猜"你的意图，越明确的约束越能把概率质量集中到你想要的输出上。
- **引导推理路径**：让模型"先想后答"（思维链）能显著提升复杂推理的准确率，因为中间推理 token 为最终答案提供了更好的"条件"。
- **示例对齐分布**：Few-shot 示例相当于在上下文里给出"任务的样本分布"，模型据此模仿格式与风格。

### 3.3 常用技巧与示例

#### （1）零样本 + 角色 + 输出约束（Zero-shot）

**反例（模糊）：**
```text
帮我写点关于咖啡的东西。
```

**正例（清晰、结构化、带约束）：**
```text
你是一位资深咖啡师。请为咖啡新手写一段介绍"手冲咖啡"的文字。
要求：
1. 语气亲切，避免专业术语；
2. 字数控制在 150 字以内；
3. 结尾给出一句鼓励尝试的话。
```

#### （2）少样本学习（Few-shot）——用示例定义任务

适合"格式固定但难以用文字描述清楚"的任务，例如情感分类：
```text
请判断下面评论的情感，只输出"正面""负面""中性"之一。

评论：这家店服务太差了，再也不来了。
情感：负面

评论：东西还行，价格也合理。
情感：中性

评论：太惊喜了，超出预期！
情感：
```
模型会顺着示例输出 `正面`。

#### （3）思维链（Chain-of-Thought, CoT）——让模型先推理再回答

**普通提问（容易算错）：**
```text
食堂有 23 个苹果，午餐用掉 20 个，又买了 6 个，现在有几个？
```

**加上 CoT 触发语：**
```text
食堂有 23 个苹果，午餐用掉 20 个，又买了 6 个，现在有几个？
请一步一步推理，最后用"答案："给出结果。
```
模型会输出：
```text
23 - 20 = 3
3 + 6 = 9
答案：9
```

#### （4）结构化输出约束（让结果可被程序解析）

```text
从下面这段文字中抽取信息，严格按 JSON 输出，不要附加任何解释：
{
  "姓名": string,
  "公司": string,
  "职位": string
}

文字：张伟是字节跳动的高级算法工程师。
```
期望输出：
```json
{ "姓名": "张伟", "公司": "字节跳动", "职位": "高级算法工程师" }
```

#### （5）分隔符与分区——避免指令和数据混淆

```text
请总结【三个反引号之间】的文章，输出不超过 3 个要点。

```
（此处放入待总结的长文……）
```
```

> 经验法则：**指令在前、数据在后，并用清晰的分隔符隔开**，可有效防止"提示词注入"和指令被数据稀释。

## 4. 上下文工程（Context Engineering）

### 4.1 核心思想

> **在有限的上下文窗口内，于"恰当的时机"为模型动态地装配"恰当的信息"——不多、不少、不旧、不乱。**

如果说提示词工程关心"指令本身"，那么上下文工程关心的是**整个进入模型的信息流**：系统提示、对话历史、检索到的知识、工具返回结果、用户输入……如何选择、压缩、排序、注入。

### 4.2 基本原理

直接对应第 2 节的底层原理：

- 上下文窗口**有限** → 必须**筛选与压缩**信息，把 token 预算花在刀刃上。
- **中间遗失**效应 → 把最关键的信息放在**开头或结尾**。
- 模型**无记忆** → 需要外部的**记忆/检索系统**把历史与知识"喂"回来。
- 知识**会过时** → 用 **RAG（检索增强生成）** 注入实时、私有数据。

一个常见的"上下文预算"心智模型：

```
[系统指令/角色]  +  [长期记忆/用户画像]  +  [检索到的相关知识(RAG)]
        +  [近期对话摘要]  +  [最近若干轮原文]  +  [当前用户输入]
        ≤  上下文窗口上限
```

### 4.3 关键技术与示例

#### （1）检索增强生成（RAG）——给模型"开卷考试"

原理：先把私有/实时文档切块、向量化存入向量库；查询时检索最相关的片段，拼进提示词再让模型作答。

```text
你是企业知识库助手。请仅依据【参考资料】回答问题；
若资料中找不到答案，请回答"资料中未提及"，不要编造。

【参考资料】
1. 公司年假政策：入职满 1 年享 5 天，满 10 年享 10 天……
2. 报销流程：需在 OA 提交发票，主管审批后 3 个工作日到账……

【问题】
入职满一年有几天年假？
```
作用：把"模型记不住/不知道"的私有知识，临时变成它"看得到"的上下文，**显著降低幻觉**。

#### （2）对话历史管理与摘要——对抗窗口上限

当多轮对话变长，不能无脑把全部历史塞进去。常见策略：

- **滑动窗口**：只保留最近 N 轮原文；
- **滚动摘要**：把更早的对话压缩成一段摘要，连同最近几轮一起送入。

```text
【对话摘要】用户在咨询一款笔记本电脑，预算 6000 元，看重续航和轻薄，
已排除游戏本，目前在 A、B 两款之间犹豫。

【最近对话】
用户：B 这款的电池容量是多少？
助手：……
用户：那续航能到几个小时？
```

#### （3）信息排序与"关键信息靠两端"

利用位置效应，把最重要的约束放最前或最后：
```text
（系统提示开头）最重要规则：任何情况下都不得透露用户的手机号。
……（中间是大量背景资料）……
（结尾再次强调）再次提醒：回答中绝不能出现手机号。
```

#### （4）工具/函数调用结果回填（Agent 场景）

Agent 会调用工具（搜索、数据库、计算器），并把**工具返回结果作为新的上下文**继续推理：
```text
用户：北京今天适合骑车吗？
→ 模型决定调用 weather(city="北京")
→ 工具返回：{"温度": 22, "天气": "晴", "风力": "3级", "AQI": 45}
→ 模型基于该结果回答：今天北京晴、22℃、微风、空气良好，非常适合骑车。
```
这里"天气数据"就是被动态装配进上下文的关键信息。

#### （5）上下文压缩与去噪

- 检索回来的长文档先做**抽取式/生成式摘要**再注入；
- 去掉与当前问题无关的段落，避免稀释注意力；
- 对重复信息去重，节省 token 预算。

## 5. 两者的关系与区别

```
提示词工程  ⊂  上下文工程（在工程实践中，前者常被视为后者的一个子集）
```

- **提示词工程**聚焦"那段你亲手写的指令文本"，是**静态、手工、面向单次交互**的优化。
- **上下文工程**聚焦"最终进入模型的完整 token 序列如何被动态构建"，是**动态、系统化、面向整个应用生命周期**的优化。
- 在简单场景（一次性问答），写好提示词就够了；在复杂场景（长对话、知识库、Agent），必须做上下文工程，而提示词只是其中一环。

| 对比项 | 提示词工程 | 上下文工程 |
| --- | --- | --- |
| 输入是否动态 | 多为静态/模板 | 动态拼装、按需检索 |
| 主要应对的原理 | 歧义、推理引导 | 窗口有限、无记忆、知识过时、位置效应 |
| 典型产物 | 一段 Prompt 模板 | 检索管线 + 记忆系统 + Prompt 模板 + 编排逻辑 |
| 谁来做 | 任何用户 | 开发者 / 系统设计者 |

## 6. 综合实战示例：构建一个客服问答助手

把两门工程结合起来，看一个端到端的例子。

**目标**：基于公司文档，回答用户售后问题，多轮对话，不能编造。

**第 1 步——系统提示（提示词工程）**
```text
你是「示例科技」的官方客服助手。
规则：
1. 仅根据【知识库片段】作答，找不到就说"这个问题我需要帮您转接人工"。
2. 语气礼貌、简洁，每次回答不超过 120 字。
3. 涉及退款金额时，必须复述订单号以供用户确认。
```

**第 2 步——动态装配上下文（上下文工程）**

每次用户提问时，后台执行：
1. 用用户问题去**向量检索**知识库，取 Top-3 相关片段；
2. 取最近 3 轮对话原文 + 更早历史的**摘要**；
3. 按"系统提示 → 知识库片段 → 历史摘要 → 最近对话 → 当前问题"的顺序拼装。

最终送入模型的上下文（示意）：
```text
[系统提示：见上]

【知识库片段】
- 退货政策：签收 7 天内无理由退货，需保持包装完整……
- 退款时效：原路退回，1-5 个工作日到账……

【历史摘要】用户购买了一台空气净化器，咨询退货事宜，订单号 SN20260601。

【最近对话】
用户：我想退货。
助手：好的，请问您的订单号是？
用户：SN20260601，能退吗？

【请回答用户最后的问题】
```

**第 3 步——模型输出**
```text
您好，订单 SN20260601 在签收 7 天内可无理由退货（需包装完整）。
确认后我为您发起，退款将原路退回，1-5 个工作日到账。
```

可以看到：**提示词工程**保证了语气、规则、格式；**上下文工程**保证了模型"看到"了正确的政策和对话历史。二者协同，才得到既准确又可靠的回答。

## 7. 常见误区与最佳实践清单

**提示词工程**

- ✅ 指令具体、可度量（"150 字以内"优于"简短一些"）。
- ✅ 复杂任务用思维链或拆分成多步。
- ✅ 需要程序消费时，强制结构化输出（JSON/表格）。
- ✅ 用分隔符隔离"指令"与"数据"，防注入。
- ❌ 不要在一个提示里堆叠互相冲突的要求。
- ❌ 不要假设模型"懂你的言外之意"。

**上下文工程**

- ✅ 把 token 预算当成稀缺资源去分配。
- ✅ 关键信息放上下文的**开头或结尾**。
- ✅ 长文档先摘要/抽取再注入，避免噪声。
- ✅ 长对话用"摘要 + 最近窗口"组合管理历史。
- ✅ 用 RAG 注入私有/实时知识，并要求模型"无依据不作答"。
- ❌ 不要无脑把所有历史/文档塞满窗口（既贵又触发"中间遗失"）。
- ❌ 不要把检索结果原样灌入而不做相关性过滤。
